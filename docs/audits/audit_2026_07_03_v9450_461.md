## Findings (severity-ordered)

### HIGH-1 — strava-webhook: unauthenticated enqueue flooding → app-wide Strava sync DoS
**File:** `supabase/functions/strava-webhook/index.ts:65-78`
(+ `enqueue_strava_backfill` = plain `pgmq.send`, `supabase/migrations/20260427_pgmq_queues.sql:191`;
global rate counter `strava_rate_state` id=1, `:94-105` and worker `strava-backfill-worker/index.ts:13-14,139-155`)

The POST handler enqueues a backfill message for **any** `owner_id` that maps to a
connected user, with **no rate limit, no dedup, and no per-user cap**. The endpoint is
public (`--no-verify-jwt`) and Strava webhook events are unsigned, so the body is fully
attacker-controlled. `enqueue_strava_backfill` is a bare `pgmq.send` (no idempotency).

Amplifier: the backfill worker throttles Strava API calls against a **single global row**
(`strava_rate_state`, `id INT PRIMARY KEY CHECK (id = 1)`, `MAX_REQUESTS = 90` / 15 min) —
shared across the entire app, not per user.

**Failure scenario:** Strava athlete IDs are public and enumerable (the founder is a
public rower; their `owner_id` is discoverable). An attacker POSTs a flood of
`{"object_type":"activity","aspect_type":"create","owner_id":<connected_id>}`. Each request
appends a `strava_backfill` message. The queue grows unbounded (attacker adds faster than
the 2-min/5-msg worker drains), the pgmq table bloats (storage/cost), and every drained
attacker message consumes the **global** 90/15-min Strava budget → **all** legitimate users'
backfills and post-connect syncs starve for as long as the attacker keeps posting.

**Suggested fix:** before enqueuing, dedup/throttle per user — e.g. skip if a `webhook`-source
backfill for that `user_id` was enqueued in the last N seconds (a short-TTL marker row or a
`SELECT` against recent queue depth), and/or dedup on `object_id`. Optionally cap
`strava_backfill` depth per user. The webhook only needs to enqueue at most one small
recent-window import per athlete per minute.

### MED-1 — strava-webhook: spoofed athlete-deauthorization forces any user into an error/reconnect state
**File:** `supabase/functions/strava-webhook/index.ts:59-64`

On `object_type:"athlete"` with `updates.authorized === "false"`, the handler writes
`sync_status="error"` + `last_error="Strava access was revoked … reconnect"` to the victim's
`strava_tokens` row. The event is never verified (unsigned, public endpoint). This directly
contradicts the header comment claiming "no attacker-controlled data reaches the DB."

**Failure scenario:** attacker POSTs
`{"object_type":"athlete","owner_id":<victim_id>,"updates":{"authorized":"false"}}`. The
victim's connection flips to `failing`; `classifyStravaSync` surfaces the red "Strava sync is
failing — re-authenticate" banner even though the athlete never revoked anything. Repeatable
at will. Damage is bounded (a manual Sync or reconnect clears it; tokens are not deleted), so
this is griefing/false-alarm rather than data loss — but it is an unauthenticated write of
attacker-chosen state to a user row.

**Suggested fix:** do not act destructively on a webhook deauth event. Instead mark a soft
flag and let the next token refresh confirm revocation (a genuinely revoked token fails
`refresh_token` → the worker already records the real error at
`strava-backfill-worker/index.ts:214-224`). Or verify by calling Strava with the stored token
before writing an error state.

### LOW-1 — GET handshake token comparison is not constant-time
**File:** `supabase/functions/strava-webhook/index.ts:31` (`token === VERIFY_TOKEN`)

String `===` short-circuits on first mismatch. In principle a network timing side-channel
could leak the verify token byte-by-byte. Practically infeasible over the public internet
(jitter ≫ signal) and low-impact (the token only gates the one-time subscription handshake;
creating a subscription also requires the Strava client secret). Note only — use a
constant-time compare if hardening. The empty-token case already fails closed
(`&& VERIFY_TOKEN`), which is correct.

### LOW-2 — scope-error connection shows "SYNC NOW" as the primary banner action (won't fix it)
**File:** `src/components/TodayView.jsx` (strava banner, `handleBannerSync`) +
`stravaSyncHealth.js:86-97`

A read-only (`activity:read_all` missing) grant is recorded as `sync_status="error"` by
`strava-oauth` (`index.ts:232-233`), which `classifyStravaSync` maps to `failing`. The
TodayView banner then offers **SYNC NOW** as the primary button, but a plain-`read` token can
never import activities — only a forced re-consent fixes it. The re-consent path exists and is
correct at OAuth-callback time (`useAppState.js` F5b + `StravaConnect` RECONNECT uses
`{force:true}`), so this is a secondary-surface UX gap, not a correctness/security bug. Consider
routing the `failing`-with-scope case to RECONNECT rather than SYNC NOW.

## Checked and found sound

- **parse-activity auth fix (v9.459 F1) is complete.** `auth.getUser()` (signature-verified)
  is the sole identity source; the job is fetched through the RLS user client scoped by
  `user_id`; all writes use the verified `user.id`. No unsigned `atob` path remains.
  (`parse-activity/index.ts:230-244`)
- **parse-activity dedup (F2) is sound.** Content-stable `external_id = upload:<sha256(bytes)>`,
  an explicit pre-insert lookup returning `duplicate:true`, the partial unique index as a
  concurrency backstop, plus an atomic job claim (`UPDATE … NOT status IN (parsing,done)`) that
  closes the check-then-act race. (`:250-334`)
- **strava-oauth sync/connect state machine cannot wedge in `syncing`.** Every return path in
  the `sync` action calls `setSyncDone(...)` (idle or error) before returning; `connect` sets
  `syncing` then the worker/first-page flips it to idle+`last_sync_at`. A backfill-only import
  that never runs self-heals via the client reconcile (`shouldReconcileStrava` → idempotent
  `sync`). (`strava-oauth/index.ts:295-408`, `312-319`)
- **Client self-heal reconcile cannot loop or double-import.** Gated by a per-session `useRef`
  (`stravaReconciledRef`) set before the async call, the `triggerStravaSync` module mutex
  (`_syncPromise`), and the DB upsert on `(user_id, external_id)`. The `log`-out-of-deps eslint
  suppression is benign: `shouldReconcileStrava` also requires `last_sync_at == null`, so a
  hydration-lag empty `log` at worst triggers one redundant idempotent sync. (`strava.js:315-319`,
  `TodayView.jsx` reconcile effect)
- **Post-connect 401 retry is bounded and correct.** `triggerStravaSyncWithRetry` retries only
  401/403 (session race), 3 attempts, 1s/3s backoff, and reuses the sync mutex.
  (`strava.js:334-353`)
- **exchangeStravaCode retry policy is correct.** 4xx not retried (single-use code →
  invalid_grant on reuse); only network/pre-flight and 5xx retried. (`strava.js:261-294`)
- **maxHR / duration fallbacks are honest and clamped.** `resolveProfileMaxHR` validates
  `maxhr ∈ [60,280]` then `220−age` for `age ∈ [5,120]`, else null → `estimateZones` returns
  null ("unknown") instead of fabricating a distribution; duration falls back to `elapsed_time`.
  (`strava-oauth/index.ts:70-89,362-371`, worker `:55-74,275-282`)
- **Rowing/Kayaking/Canoeing → "row" mapping** is consistent across edge oauth, worker, and the
  client transform. (`strava-oauth/index.ts:41`, worker `:28`, `strava.js:17`)
- **Backfill worker robustness:** poison ceiling dead-letters after `MAX_READ=10` reads;
  revoked/401/403 tokens write `last_error` before dropping the message (no silent "healthy");
  empty page and imported page both set `last_sync_at`. (`strava-backfill-worker/index.ts:175-179,
  214-251,254-322`)
- **strava-backfill-worker is not the public surface** — it requires `isVerifiedServiceCall`
  (constant-time shared secret), unlike the webhook. (`:128-130`)
- **Client direct-import path in strava.js is deliberately unwired** (documented in the file
  header + StravaConnect); not re-wired. Not a finding.
- **Billing-webhook restore (v9.458, migration 20260634) correctly reapplies both guards** —
  rejects `no_event_id` and requires `tier IN ('coach','club')` on the current function body;
  amount parsing is regex-guarded. (`20260634_restore_subscription_event_hardening.sql`)
- **XSS guard (v9.457) is sound** — `renderSnippet` escapes `& < >` first, then restores only
  `<b>…</b>` → `<mark>`. (`GlobalSearch.jsx:42-50`)
- **`never_synced` health state** is correctly distinguished from `stale`/`failing` and only
  applies within `STRAVA_NEVER_SYNCED_FRESH_DAYS` of `updated_at`, falling through to `stale`
  for old never-synced tokens. (`stravaSyncHealth.js:99-117`)
