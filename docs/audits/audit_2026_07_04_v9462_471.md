## HIGH-1 — Summary re-import permanently clobbers streams enrichment (marker blocks re-enrich)

**Files:** `supabase/functions/_shared/stravaActivity.ts:197-274` (buildTrainingLogRow),
`supabase/functions/strava-backfill-worker/index.ts:339-347` (page upsert),
`supabase/functions/strava-oauth/index.ts:298-311` (sync upsert),
`supabase/functions/_shared/stravaActivity.ts:169-174` (enqueue filter `.is("stream_enriched_at", null)`).

**Scenario.** The v9.466 enrich pass writes stream-derived values onto the row: true
5-zone distribution (`zones`), stream NP (`np`), power-TSS from stream NP (`tss`),
athlete-entered `rpe` + `rpe_method='athlete'` from `perceived_exertion` — and stamps
`stream_enriched_at`. Any later **summary** upsert of the same activity — a webhook
`update` event (rename/type/privacy change → 2-day window re-import) or, worse, any
manual **SYNC NOW** (30-day window, every activity in it) — runs
`buildTrainingLogRow` and `upsert(row, { onConflict: "user_id,external_id" })`, which
sets ALL payload columns:

- `zones` reverts to the single-band `estimateZones` smear (line 257),
- `np` reverts to summary `weighted_average_watts` (line 227),
- `rpe`/`rpe_method` revert to `derived_hr` — the athlete's reported perceived
  exertion is overwritten by an estimate (lines 239, 272),
- `tss` reverts to summary-NP power-TSS or TRIMP (line 235).

`stream_enriched_at` is not in the payload, so it **stays set** — and
`enqueueStreamEnrichment` filters `.is("stream_enriched_at", null)`, so the activity is
**never re-enriched**. The P1 feature silently self-destructs for the entire 30-day
window on every Sync tap. (`decoupling_pct`, `w_prime_exhausted`, `calories` survive —
they are not in the summary payload.)

The design's own risk item (strava_data_enhancements_2026_07_03.md, risk 5) required
"all derived fields are deterministic functions of the payload" — the streams-derived
fields are functions of the *streams* payload, which the summary upsert does not carry.
The condition is violated, not satisfied.

**Severity:** HIGH (silent, routine-trigger, permanent data-quality regression negating v9.466).

**Suggested fix (pick one):**
1. Before the page/sync upsert, select the batch's `external_id`s where
   `stream_enriched_at IS NOT NULL` and strip the contested columns
   (`zones,np,tss,rpe,rpe_method`) from those rows' payloads (needs per-row payload
   shapes → see MED-4 in dataMigration for PostgREST uniform-key caveat; simplest is
   two upsert batches: full-shape for unenriched, reduced-shape for enriched).
2. Or set `stream_enriched_at: null` in `buildTrainingLogRow` output so a re-import
   re-enqueues enrichment (costs 2 API calls per touched activity — 30-day window ×
   sync taps can be expensive; budget-guarded but wasteful).
3. Or move the merge into an RPC that upserts with
   `SET col = CASE WHEN training_log.stream_enriched_at IS NULL THEN EXCLUDED.col ELSE training_log.col END`.

Option 1/3 preferred; option 3 is the only race-free one.

## HIGH-2 — TrainingLog inline edit strips every metric/enrichment field (locally AND in DB)

**Files:** `src/components/TrainingLog.jsx:309-321` (add/save), `:350-362` (startEdit),
`src/hooks/useSupabaseData.js:99-139` (logEntryToRow), `:312-315` (changed → update).

**Scenario.** `startEdit(s)` copies only `date/type/duration/rpe/notes(/zones)` into the
form. On save, `raw = { id, ...form, duration, rpe, tss, zones }` — the original entry
is **not** spread in. `sanitizeLogEntry(raw)` can only preserve fields present on its
input, so the replacement entry has no `avgHR, avgCadence, distanceM, np, avgPower,
maxHR, elevationGainM, kilojoules, sufferScore, startTime, rpeMethod, decouplingPct,
wPrimeExhausted, wPrimeMethod, calories, source`. The diff-by-id sync then runs
`update(logEntryToRow(entry))`, which maps every absent field to an explicit `null`
(posInt(undefined) → null, etc.) and hardcodes `source:'manual'` — so fixing a typo in
the notes of an enriched Strava ride **NULLs all enrichment columns in the DB**, drops
the STR badge (`source` flips to 'manual'), and removes the `~` derived-RPE marker.
Because `stream_enriched_at` stays set, the streams-derived fields are not re-derivable
(same dead-end as HIGH-1; summary fields come back only if a webhook/sync re-import
covers the activity's window).

This directly contradicts the v9.465 changelog claim that logEntryToRow's round-trip
means "an edited Strava/FIT entry doesn't wipe them on the diff-by-id sync upsert" —
the round-trip protects entries edited *elsewhere* (e.g. bulk-tag at :368, which spreads
`{ ...e, tags }`), but the primary edit path discards the fields before the mapper ever
sees them. (The avgHR/distanceM subset of this bug predates the audited range —
v9.397 — but the audited versions both widened the blast radius by 11 columns and
shipped the incorrect safety claim.)

**Severity:** HIGH (user-triggered permanent data loss on the flagship enrichment data).

**Suggested fix:** in `add()` when `editingId !== null`, merge over the original:
`const orig = log.find(e => e.id === editingId); const raw = { ...orig, ...form, duration:…, rpe:…, tss, … }`
— and preserve `orig.source` (don't let `logEntryToRow`'s hardcoded `'manual'` overwrite
a 'strava'/'fit' source: make `source: entry.source || 'manual'` in the mapper, or keep
the hardcode only for the dataMigration call site, which relies on it).

## MED-1 — needsReconnect (v9.470) misses the manual-sync 401 path → RECONNECT routing defeated

**Files:** `supabase/functions/strava-oauth/index.ts:94-97` (fetchActivitiesPage throw),
`:280-285` (setSyncDone(msg)), `src/lib/athlete/stravaSyncHealth.js:94` (regex).

**Scenario.** A token revoked at Strava while still unexpired: `refreshIfExpired`
returns early (not within 5 min of expiry, `strava-oauth/index.ts` early `return null`),
the activities fetch 401s, and `fetchActivitiesPage` throws
`` `Strava API ${resp.status}: ${body}` `` → `last_error = "Strava API 401: {"message":"Authorization Error",…}"`.
That string matches neither `revoked`, `rejected`, nor `read-only`, so
`needsReconnect=false` and the TodayView banner offers **SYNC NOW** — the
guaranteed-failure loop v9.470 was shipped to kill. Each tap rewrites the same
non-matching string, so the state never converges to the worker's matching
"authorization rejected" string unless a queued backfill message happens to exist.

**Severity:** MED (the exact LOW-2 loop persists on the most user-visible retry path).

**Suggested fix:** in the sync branch, mirror the worker: on `Strava API 401/403`, write
the canonical `"Strava authorization rejected — please reconnect Strava"` string
(strava-backfill-worker/index.ts:312) instead of the raw thrown message. Fixing the
producer is better than widening the client regex (per the v9.470 DEPENDS-ON note, the
strings are the contract).

## MED-2 — Enrich stream handling: independent null-filtering breaks series alignment; `time` stream fetched but unused

**Files:** `supabase/functions/strava-backfill-worker/index.ts:235-241` (asSeries),
`:275` (decoupling pairing), `supabase/functions/_shared/streamScience.ts:30-46, 70-85`.

Two related failure modes, both absent from the FIT path it claims parity with:

1. **Alignment.** `asSeries` filters `typeof v === "number"` per stream *independently*.
   Strava stream arrays contain `null` gap samples (esp. `watts` and `heartrate`).
   parse-activity keeps arrays index-aligned by 0-filling
   (`parse-activity/index.ts:98-103`); the worker instead compacts each stream, so
   `hr[i]` and `watts[i]` stop referring to the same moment. `decouplingPct(hr, watts)`
   then splits halves by `usableHR.length` while slicing the *shorter* effort series by
   the same indices — `eff.slice(mid)` can be short or empty, `avgEff → 0.001` fallback,
   and the stored `decoupling_pct` becomes an absurd value (≈±100%), which
   `decouplingTrend` happily averages. NP/W′ on compacted watts also compress time
   (dropped null-seconds should be 0 W ⇒ depletion/recovery seconds vanish).
2. **1-Hz assumption.** `time` is requested in the keys but never read. For
   smart-recording devices (samples every 3–7 s) and auto-pause gaps, 30 *samples* ≠ 30 s:
   NP's rolling window over-smooths, `wPrimeExhausted` under-counts recovery seconds
   (`exp(-1/tau)` applied once per sample, not per elapsed second), and decoupling halves
   split by sample count, not time. parse-activity has the same assumption on FIT
   records, so this is "FIT parity" in the loose sense — but the time stream is already
   being paid for in the same API call.

**Severity:** MED (1 is a correctness bug writing garbage decoupling on gappy streams;
2 is a bounded accuracy tradeoff, partially precedented).

**Suggested fix:** in `asSeries`, map non-numbers to 0 instead of filtering
(exact parse-activity semantics), which fixes alignment; optionally use the `time`
stream to expand samples to 1-Hz (repeat/zero-fill per gap) before NP/W′/decoupling.

## MED-3 — Edge `mapStravaType` missing `GravelRide` (client map has it) → gravel rides typed 'other'

**Files:** `supabase/functions/_shared/stravaActivity.ts:17-31` vs `src/lib/strava.js:9-19`.

The v9.462 "mapping parity" pass gave the (dead-code) client `SPORT_TYPE_MAP`
`GravelRide: 'Ride'`, but the live edge mapper has no `GravelRide` key → `"other"`.
A gravel ride imports with `type:'other'`: excluded from `hasCyclingData` gating

**Severity:** MED-LOW (data quality; silent for gravel-primary athletes).
**Fix:** add `GravelRide: "bike"` (and consider `Velomobile`/`Handcycle` explicitly out).

## MED-4 — dataMigration batch insert fails on mixed uuid/non-uuid entry ids (heterogeneous keys)

**Files:** `src/lib/dataMigration.js:90, 109-112`, `src/hooks/useSupabaseData.js:107`.

`logEntryToRow` includes `id` **conditionally** (`...(isUuid(entry.id) ? { id } : {})`).
A migration batch mixing uuid-id entries with non-uuid-string-id entries (the
`newId()` fallback format `"<ts>-<rand>"` from a crypto-less browser, or backup-JSON
imports carrying foreign string ids — `migrateLogIdsToUuid` deliberately keeps any
non-empty string id, validate.js:177) produces rows with *different key sets* in one
`insert(rows.slice(i, i+100))`. PostgREST builds the column list from the union: rows
missing `id` get `id=null` → `23502 not-null violation` → the **whole 100-row batch
fails**, `errors` throws, `MIGRATED_KEY` never set, and Retry hits the identical error
forever (cleanup+reinsert loop; no data loss, but migration is permanently stuck).
The pre-v9.468 hand-built row never included `id`, so the column stayed out of the
INSERT entirely and the default applied — this is a v9.468 regression window.

**Severity:** MED-LOW (needs a mixed-id guest log — rare but the failure is total and
sticky when hit; migration is the single most data-sensitive flow).
**Fix:** in dataMigration, normalize the batch: `rows.map(r => ({ id: r.id ?? crypto.randomUUID(), ...r }))`
or strip `id` from all rows when any row lacks it (uuid pass-through was a nice-to-have).

## LOW-1 — Editing a null-RPE entry resurrects the rpe-0 fabrication locally

**Files:** `src/components/TrainingLog.jsx:356` (`String(entry.rpe)` → `"null"`),
`:307,313` (`parseInt("null")` → NaN), `src/lib/validate.js:75,91` (clamp(NaN)→0).

`startEdit` on a null-rpe entry sets the controlled `<select value="null">` (no matching
option → renders unselected). Saving without touching it yields `rpe: NaN` →
sanitizeLogEntry's null-guard checks `== null || === ''` — NaN passes through to
`clamp` → **0**. Locally the entry now has `rpe: 0` (the fabrication v9.469 killed);
the DB stays honest only by luck (`Number(0)||null` → null in logEntryToRow), so local
and server diverge until rehydration. The fixed averages filter `> 0`, so numbers stay
right; the log cell shows a green `0`.
**Fix:** `rpe: entry.rpe == null ? '' : String(entry.rpe)` in startEdit + treat
`''`/NaN as null in `add()` (and consider `clamp` lower bound 1, since
`logEntryToRow` already refuses to persist 0 — see LOW-6).

## LOW-2 — v9.469 census misses: two more `||5`-inside-aggregation sites + a `≤`-gate that null passes

**Files:** `src/lib/patterns.js:375`, `src/lib/intelligence.js:106`,
`src/lib/athlete/decouplingTrend.js:98-99`.

- `patterns.js:375` (`findOptimalTrainingPattern` week scoring):
  `avgRPE = sessions.reduce((s,e)=>s+(e.rpe||5),0)/n` — a ||5-diluted *average*, the
  exact class the E3 design fixed in 4 places, not a "zone-bucketing" member. Impact is
  soft (weeks are compared against each other under the same dilution) but the census
  classified it wrong by omission.
- `intelligence.js:106`: correlation pairs get `rpe: e.rpe||5` — fabricated 5s feed the
  load→recovery correlation.
- `decouplingTrend.js:98-99`: `Number(e?.rpe)` turns null into **0**, and the gate is
  `rpe > AEROBIC_RPE_CAP → exclude` — a `≤`-style gate, which null-as-0 *passes*. The
  census's "GATES: null fails the threshold" holds only for `≥` gates. Practically
  unreachable today (decouplingPct implies HR implies derived rpe), but the invariant
  is one FIT-import shape away from mattering.

**Fix:** present-only denominator at patterns.js:375; skip null-rpe pairs at
intelligence.js:106; explicit `e?.rpe == null → continue` in decouplingTrend.

## LOW-4 — MaxHrNudge setProfile uses a render-time snapshot spread

**Files:** `src/components/MaxHrNudge.jsx:23`, `src/hooks/useProfileQuery.js:100-110`.
`setProfile({ ...profile, maxhr })` passes the full stale snapshot as `next`; the
merge-on-write (`{...server, ...next}`) only protects keys *absent* from `next` — and
none are, so a concurrent cross-device edit of any other key (Device B updates ftp,
Device A taps UPDATE TO N) is clobbered with Device A's stale value. The functional
form `setProfile(p => ({ ...p, maxhr: String(hit.observedMax) }))` reads the query
cache (fresher) and is the cheap improvement; the residual race is inherent to the
blob design and shared with every existing caller (pattern-consistent, so LOW).
Also noted, deliberate per changelog: a single HR-strap artifact ≤250 bpm on a session
that has avgHR will nudge — mitigation is "the athlete decides".

## LOW-5 — Dead-lettered enrich messages for out-of-window activities are unrecoverable until reconnect

**File:** `supabase/functions/strava-backfill-worker/index.ts:20,122-126,219-222`.
The v9.468 transient-5xx path leaves the message for VT retry; at `read_ct ≥ 10`
(~20 min of consecutive failures) it dead-letters. `stream_enriched_at` stays null —
correct — but re-enqueue only happens when a page import re-covers the activity
(webhook = 2-day window, sync = 30-day). An older activity whose enrich died in a long
Strava outage is silently never enriched until a full re-backfill (reconnect). Accepted
poison/VT tradeoff; worth a `console.warn` distinguishing "dead-lettered enrich for
external_id X" so the operator can see it (current warn logs only msg id).

## LOW-6 — rpe 0 cannot round-trip; sanitizer and mapper disagree on the floor

**Files:** `src/hooks/useSupabaseData.js:113` (`Number(entry.rpe)||null`),
`src/lib/validate.js:91` (`clamp(e.rpe, 0, 10)`).
The sanitizer accepts 0 (and coerces negatives/garbage to 0) but the row mapper drops 0
to null, so a local rpe-0 entry hydrates back as null. No form can emit 0 (selects are
1–10, QuickAdd default 6), so this is only reachable via imports/the LOW-1 NaN path —
but the two layers should agree: make the sanitizer floor 1, or map 0 explicitly.

## Notes / pre-existing observations (out of audited range, recorded for the backlog)
- `estimateTSS` (TRIMP×1.2, no LTHR) vs parse-activity's LTHR-normalized TRIMP are
  different scales — Strava-summary TSS and FIT TSS for the same session diverge.
  Pre-v9.462 behavior; becomes moot per-activity once streams enrichment writes
  power-TSS, but HR-only athletes see the mismatch.
- useSyncedTable hydration caps training_log at 365 d / 1000 rows and *replaces* the
  local array — local-only entries older than the window disappear from view after
  login (no deletes fire, server data intact). Pre-existing shape.
