# Program ↔ Training-History Data-Flow Audit — 2026-07-08

Scope: do TRAINING PROGRAMS see and use imported (Strava) history, per sport (run/row/bike)?
Canonical imported entry shape: `{ type: 'run'|'bike'|'row'|..., duration: MINUTES, tss, distanceM, avgHR?, np?, rpe: may-be-null, sessionTag, powerPeaks? }`.
Excluded (already fixed): EliteProgramCard trainDays/weeklyHours phantoms + staleness vdot (v9.485/487-488), sessionExecution num(null), plan-aware coach EXECUTION PROFILE.

Findings appended as verified. Format: severity, file:line, scenario, suggested fix.

## F1 — HIGH — findRecentBest has zero rowing support: a rowing athlete's entire imported history is invisible to the Elite Program autofill + sport default
- **File**: `src/lib/athlete/recentBest.js:18-26` (RUN_RE/BIKE_RE/SWIM_RE + BUCKETS have no row entry), consumed at `src/components/dashboard/EliteProgramCard.jsx:1728-1735`.
- **Scenario (founder)**: all 28 imported entries are `type:'row'` with `distanceM`/`duration`. `classifySport()` returns null for every one (regexes only match run/bike/swim; BUCKETS has no 2000/5000/6000m rowing buckets). `findRecentBest` → null, so (a) the "USE MY RECENT BEST" chip never appears for the sport the orchestrator itself fully supports (`sport:'rowing'`, `rowingGainPerBlock`, `predict2000m`, rowing sample weeks), and (b) `defaultSport` falls through `recentBest?.sport || 'run'` — a rower without `primarySport` set gets a RUN program form pre-selected. The imported history exists, is correctly shaped, and is simply never consulted.
- **Fix**: add `ROW_RE = /row|erg/i` (before run per the v9.487 F11 lesson — although 'row' happens not to collide with /run/, ordering discipline is free) + `BUCKETS.rowing = [2000, 5000, 6000, 10000]`, extend the `bestPerBucket`/`sportCount` maps and the sport preference order.

## F2 — MED — Elite-program "MAKE-UP" nudge can't see Strava-imported sessions: false "you missed your key session" for athletes who did the workout
- **File**: `src/components/dashboard/TodayProgrammedSessionCard.jsx:33-40` (`logIntentKey`) + `:194-209` (match loop).
- **Scenario (founder)**: program prescribes Thu "AT threshold 4x2000m". Founder does it on the erg; webhook sync imports it as `{ type:'row', notes:'Afternoon Rowing · 12.00 km · avg HR 152', session_tag:'moderate' }` (see `supabase/functions/_shared/stravaActivity.ts:323-344` — notes = activity name + km + HR; session_tag vocabulary is junk/recovery/moderate/unplanned_high, never threshold/long/intervals). `logIntentKey` greps type+intent+notes+session for `/long|threshold|tempo|cruise|interval|vo2|race-pace/` → null ≠ 'threshold' → `matched=false` → the card tells him he MISSED the threshold session he completed. Fires for every Strava-first logger whose activity names aren't English workout jargon; only suppressed if the athlete happens to title the activity "Threshold ...".
- **Note**: `logIntentKey` also reads `entry.session` — no such field exists on any entry shape (phantom); the imported `sessionTag` field is NOT read (and wouldn't help given its vocabulary).
- **Fix**: fall back to load-based matching — an entry on the target date with duration ≥ ~70% of the planned key session's `durationMin` (or `rpe>=6` / tss within band) should count as done; keyword match can stay as the fast path. Also drop the phantom `entry.session` read.

## F3 — LOW — rowing "2k pace" key sessions are invisible to the make-up detector by vocabulary
- **File**: `src/components/dashboard/TodayProgrammedSessionCard.jsx:22-31` (`makeupKeyIntent`).
- **Scenario**: rowing sample-week key intents "TR pieces 5x1000m" (Build), "2k pace 8x500m" / "AN power 10x250m" (Peak) match none of `/long|threshold|tempo|cruise|interval|vo2|race-pace/` → these key days can never generate a make-up nudge (silent no-op, not a false positive — hence LOW). The nudge feature is effectively run/bike-vocabulary-only.
- **Fix**: extend the intent regexes with rowing vocabulary (`/2k pace|tr pieces|an power|at (threshold|pieces)/`) or match on the sample day's Z4+Z5 minutes instead of label text.
## F4 — HIGH — Recording a field test WIPES the elite program: FieldTestModal persists the wrong shape to `sporeus-eliteProgram`
- **File**: `src/components/FieldTestModal.jsx:172-174` (`setPersistedProgram(reAnchored)`), vs the `{ input, form }` contract at `src/components/dashboard/EliteProgramCard.jsx:1737-1738` (`if (!persisted?.input) return { result: null … }`, save site `:1837 setPersisted({ input: enriched, form })`) and `src/components/dashboard/TodayProgrammedSessionCard.jsx:127-128` (`if (!persisted?.input) return null`).
- **Scenario (founder)**: rowing program is running; after Base he records his 2k erg test in FieldTestModal (the ONLY path through which new tested physiology reaches the program — mission Q3). `reAnchorEliteProgram` correctly rebuilds (rowing units are right on this path: split sec/500m × 4 → 2k time, `eliteProgram.js:1692-1698`), then the modal writes the raw PROGRAM OBJECT into `sporeus-eliteProgram`. That object has no `.input` key, so on next render EliteProgramCard falls back to the empty generation form and TodayProgrammedSessionCard returns null — the program (and the daily answer) vanish. Every consumer of this storage key rebuilds from `persisted.input`; none accepts a program-shaped value. Recording the field test destroys the program instead of adapting it.
- Also: the re-anchored TSS uses fresh CTL correctly — `ProgramCalendar.jsx:434` passes `profile={{ currentCTL: Math.max(20, calcLoad(log)?.ctl ?? 0) }}`, so imported history DOES anchor the rebuild (checked-clean on that sub-question) — but the result is then persisted uselessly.
- **Fix**: persist `{ input: <rebuilt input equivalent> , form: persisted.form }` — i.e. store the NEW `currentPR`/`raceDate`/`profile` input that reAnchor derived (`newCurrentPR` is already computed inside `reAnchorEliteProgram`; return it in `reAnchored.input` and write that), or teach the readers to accept both shapes.

## F5 — LOW — FieldTestModal before/after TSS comparison always reads 0 vs 0
- **File**: `src/components/FieldTestModal.jsx:61-71` (`sumTSS` does `weeklyTSS.find(w => w.week === wkNum)` + `w.tss`).
- **Scenario**: `buildEliteProgram().weeklyTSS` is an array of NUMBERS (`eliteProgram.js:1415-1420`), not `{week, tss}` objects (only the cycle-gate overlay produces objects, and even those aren't keyed by `week`). `find(w => w.week === …)` never matches → `prevTSS = newTSS = 0` → the "before/after Peak+Taper TSS" panel the modal was built to show is dead for every sport.
- **Fix**: index by week number: `total += Number(weeklyTSS[wkNum - 1]?.tss ?? weeklyTSS[wkNum - 1]) || 0`.

## F6 — MED — `_logSport.js` has no rowing class: rowing-program compliance works only by the null-passthrough accident, and the program's own cross-train days are EXCLUDED from compliance
- **File**: `src/lib/athlete/_logSport.js:15-23` (`logEntrySport` returns only run/bike/swim/triathlon/null), consumers `src/lib/athlete/calendarProgress.js:112-123`, `src/lib/athlete/planAdherence.js:309`, `src/lib/athlete/planLifecycle.js:147`.
- **Scenario (founder)**: rowing program → `programSport='rowing'`. Every imported `type:'row'` entry classifies as null → `entryMatchesProgramSport` counts it via the "unclassified passthrough". So headline adherence works — but by accident, and three real leaks follow: (1) the rowing sample weeks PRESCRIBE "Cross-train (run/bike) 45min" Sundays (`eliteProgram.js:891,907`) — the founder's imported Sunday run (`type:'run'`) classifies as 'run' ≠ 'rowing' and is FILTERED OUT of that day's calendarProgress/planAdherence, so following the plan exactly reads as a missed day + weekly TSS shortfall; (2) `sportMatched` (calendarProgress:122) is false for every rowing entry on a rowing program — currently unconsumed by UI, but any future consumer inherits the bug silently; (3) walk/strength/'other' entries also pass through as null and inflate rowing-program actual TSS (defensible for whole-body load, but it means the rowing filter effectively only filters OUT the sports a rower is told to cross-train in — the worst possible polarity).
- **Fix**: add `if (/row|erg|kayak|canoe/.test(t)) return 'rowing'` to `logEntrySport` (order before /run/ per v9.487 discipline), make `entryMatchesProgramSport` accept run/bike as cross-train credit for rowing programs on days whose blueprint intent is cross-train (or at minimum count them at entry-TSS toward weekly totals).
